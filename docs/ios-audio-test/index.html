iOS Audio Diagnostics

Build v44 (locked-screen probe) — click Run on each test, then tap ✓ Heard or ✗ Silent. Tests 11–13 are run with the phone LOCKED.

After each test fires, two buttons appear: ✓ Heard and ✗ Silent. Tap whichever is true — the result line only finalizes after you do. For tests 7 and 9 you may see "decoding… please wait" first; wait until it changes before tapping.

Locked-screen tests

These two are the important ones. They answer what screen-on tests can't: does audio keep playing — and stay on time — when the phone's screen is off in a pocket? Procedure: tap Run, and the moment you hear the FIRST beep/number, turn the screen off (press the side button once) and put it in your pocket. You do NOT need to unlock with a passcode afterward — "locked" here just means the screen is dark. Listen for ~35 seconds. You should hear 7 sounds, one every 5 seconds, rising in pitch (test 11) or counting 1–7 (test 12). Then pull the phone out, unlock, and tap what you actually heard.

Please run both twice: once in Safari, then again after adding this page to your Home Screen and opening it from there (installed-PWA mode can behave differently).
11. Continuous <audio> track survives lock
Plays ONE continuous audio file with 7 beeps baked 5s apart (the "render the whole event into one file" path). If you hear all 7 evenly spaced with the screen locked, a pre-rendered track survives the pocket.
—
12. Reactive timer + TTS cadence under lock
Fires a spoken number every 5s via a JS timer (the "live TTS fallback" path). If iOS throttles timers when locked, the numbers bunch up or stall — that's the sync risk. The +ms deltas in the log show the real firing times.
—
13. Reactive Web Audio beep via MediaStream tunnel under lock
The key one. Same every-5s timer as test 12, but instead of TTS it creates a fresh Web Audio beep each time, routed through a live <audio> stream (the WebRTC "tunnel" that may keep the audio engine alive when locked). If you hear all 7 rising beeps with the screen off, runtime-generated audio survives the pocket and we do NOT need to pre-bake one big file. Silent = the tunnel doesn't help, baked file it is.
—
14. Multi-minute tunnel — survives a long lock?
Same tunnel as test 13, stretched to ~4 minutes (a beep every 20s, 12 total). Some reports say iOS stops Web Audio after "a few minutes" locked — this is the probe that would catch that. Tap Run, hear the first beep, then turn the screen off and pocket it for the full ~4 minutes before reporting. All 12 rising beeps evenly spaced → the tunnel holds for real event lengths. Stops partway → note roughly when (the log timestamps every beep with the context state).
